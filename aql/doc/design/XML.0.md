# AQL XML Format Design

Status: design draft, no implementation.

A tree-structured TSX-flavoured format as an alternate concrete syntax
for AQL programs and data. The format follows JSX/TSX conventions —
single `{}` for embedded expressions, self-closing tags, fragments,
attribute expression values, and spread — adapted to AQL's
lowercase-word naming. It is targeted at tooling, code generation,
and embedding in document workflows where the concatenative
jsonic-style syntax is awkward. This document specifies the format,
the lexer/parser strategy (informed by JavaScript's JSX and
template-string engines), the query API (informed by CSS-selector
libraries), and the embedding mechanism for foreign syntaxes.


## 1. Why a TSX-flavoured format?

Concatenative source is dense and ergonomic for hand-written AQL but
is poorly suited to:

- Code generated by other tools (the tree shape is closer to AST).
- Hybrid documents that intersperse AQL with rich text, diagrams, or
  other DSLs.
- Language servers and editors that want a stable structural format
  to round-trip through formatters and refactors.
- Document/database workflows where AQL fragments are stored alongside
  content.

Borrowing JSX/TSX conventions — rather than inventing a new tree
syntax — gets familiarity for free: anyone who has written React
already knows how interpolation, attribute values, fragments, and
spread behave. The syntax is also supported by mature editor tooling
(highlighters, formatters, language servers, JSX-aware diff tools)
which we can reuse with minimal customisation. Where TSX and AQL
disagree on a convention (the most visible one being TSX's
"capitalised tags = component" rule, which conflicts with AQL's
lowercase-with-hyphens word convention), the doc calls out the
adaptation.


## 2. Format sketch

A program is a tree of element tags interspersed with brace-delimited
AQL expressions:

```tsx
<aql>
  <def name="double">
    <dup />
    <add />
  </def>

  <double>{21}</double>          {/* call double with literal 21 → 42 */}

  <map>
    <entry key="name">{"alice"}</entry>
    <entry key="age">{30}</entry>
  </map>
</aql>
```

### 2.1 Tag conventions

- **Structural primitives (lowercase, fixed set):** `<aql>`, `<def>`,
  `<fn>`, `<sig>`, `<list>`, `<map>`, `<entry>`, `<aql-embed>`,
  `<comment>`. These are recognised by the parser and produce the
  corresponding engine value or scaffolding.
- **Word invocations (lowercase, open set):** any other lowercase
  tag is an AQL word lookup. `<double>` calls the `double` word;
  `<sortby>` calls `sortby`. AQL's naming convention is
  lowercase-with-hyphens, so the TSX "capitalised = component" rule
  is inverted here: structural primitives have a small fixed set of
  reserved names, and everything else is a word.
- **Self-closing form:** `<word />` is the no-argument invocation,
  identical to the word appearing alone in concatenative source.
  Useful for terminal words: `<dup />`, `<add />`, `<print />`.
- **Children = forward arguments.** A tag's element and expression
  children become the word's forward args, in document order.
  `<add>{1}{2}</add>` is identical to `add 1 2`.
- **Attributes = named args / metadata.** Attributes carry small
  named values: `<def name="double">`, `<fn export>`. Values may be
  string literals (`name="double"`) or AQL expressions
  (`size={n}`). Boolean shorthand follows JSX: a bare `export` is
  equivalent to `export={true}`.
- **Fragments `<>...</>`** group sibling elements without a wrapper
  — useful at the top of a file when there is no single root, or
  inside a tag whose children should be flattened.
- **Spread `{...list}`** splats a list as forward arguments —
  the same shape as TSX prop spread, retargeted to positional args.

### 2.2 Common forms

```tsx
<dup />                            {/* word, no args */}
<double>{21}</double>              {/* word with one literal arg */}
<add>{1}{2}</add>                  {/* word with two args */}
<print>Hello, {name}!</print>      {/* interpolated tree text */}

<def name="square">                {/* def: name attribute, body children */}
  <dup />
  <mul />
</def>

<fn name="sq">                     {/* fn: signature + body */}
  <sig in={["Number"]} out={["Number"]}>
    <dup />
    <mul />
  </sig>
</fn>

<list>{1}{2}{3}</list>             {/* list literal as element */}
<map>
  <entry key="x">{10}</entry>
  <entry key="y">{20}</entry>
</map>

{[1, 2, 3]}                        {/* AQL list literal in expression mode */}

<each fn={quote dup}>{[1,2,3]}</each>  {/* higher-order: fn via attribute */}

<add {...args} />                  {/* spread args from a list */}

<>                                 {/* fragment: top-level multi-rooted */}
  <def name="a"><i>{1}</i></def>
  <def name="b"><i>{2}</i></def>
</>
```

### 2.3 Tree mode and expression mode

The format alternates between two grammatical modes:

- **Tree mode** is the TSX-like syntax: tags, attributes, child
  elements, and free text.
- **Expression mode** is normal AQL, entered via `{...}` inside
  tree mode (either as an attribute value or as a child of another
  tag) and exited by the matching `}`.

Inside an expression, opening another `<...>` tag re-enters tree
mode, and inside that tag, another `{...}` re-enters expression
mode. There is no depth limit on nesting; the lexer keeps a small
mode stack and pivots on the boundary tokens `{`, `}`, `<`, `>`.

The mapping is intentionally shallow: every concatenative token has
a corresponding TSX form, and every TSX form maps to a sequence of
concatenative tokens. Round-tripping `concat → TSX → concat` is a
mechanical walk; the only metadata that does not survive without
help is whitespace and comments (see §5).


## 3. Interpolation and expression embedding

Embedded AQL expressions use single braces, exactly as JSX/TSX does:

```tsx
<print>Hello, {name}!</print>
```

The `{...}` switches from tree mode into AQL expression mode. Inside
the braces is a normal AQL expression that produces a value; that
value is spliced into the surrounding tree as a child (or, in
attribute position, as the attribute's value).

### 3.1 Why `{}` instead of `${}`

An earlier draft of this design used JavaScript template-literal
`${}`. That is the right choice for *strings*, where every literal
`{` is a hazard (CSS, JSON, regex quantifiers, set notation all use
`{}`). Inside JSX/TSX *tree* text the trade-off is reversed:

- `<` and `>` are already reserved as tag delimiters; readers expect
  the tree language to "own" some characters.
- `{` is rare in markup-like text — most natural-language prose has
  no curly braces at all.
- Once `{` is the chosen sigil, the leading `$` is pure noise.

The escape mechanism — `{'{'}` to emit a literal brace — is the
same one TSX uses, and the same one we reach for when we need a
literal `<` or `>` in tree text. The cost is small because in
practice tree content is mostly natural-language prose or AQL
expressions, neither of which collides much with `{}`.

Backtick template strings inside an `{...}` expression keep using
`${...}` (we are back in string mode there) — the same boundary,
the same convention as JS itself. AQL's existing backtick interp
(see CLAUDE.md, "Template string interpolation") is unaffected.

### 3.2 Why single braces, not doubled `{{ ... }}`

Some templating systems (Mustache, Handlebars, Vue's text
interpolation) use `{{ ... }}` to avoid the rare collision with
literal `{}` in markup. JSX/TSX uses single `{}` for two reasons we
inherit:

1. The escape mechanism is good enough for the rare case.
2. Single braces compose cleanly with the rest of the language: a
   JSX expression is a JS expression and can carry object literals
   (`{ key: value }`) as a natural sub-form.

AQL similarly uses `{...}` for map literals at the language level.
Inside tree mode, `{...}` is unambiguously an expression boundary;
map literals nest as `{( {key: value} )}` if the outer braces would
be mistaken for the boundary — the same pattern TSX uses for
returning an object from an arrow function: `() => ({foo: 1})`.

### 3.3 Parsing strategy — single grammar, not nested parsers

JavaScript engines (V8, SpiderMonkey, JavaScriptCore) and parsers
(Babel, Acorn, esprima, swc, sucrase) all handle JSX and
template-literal interpolation as a *single grammar* rather than
re-entering a fresh parser. The lexer carries a small mode stack —
`tree`, `expr`, `tplstr` — and pivots on the boundary tokens.
Each pivot increments or decrements the relevant nesting counter so
that matching delimiters can be paired back up regardless of depth.

The advantages of the single-grammar approach over spinning up a
sub-parser:

- One token stream, one error-reporting context, one set of source
  positions.
- Nested interpolations and nested tags at arbitrary depth work for
  free.
- No marshalling between parser instances, no doubled error
  infrastructure, no boundary semantics to define.

AQL already follows this pattern for backtick template strings —
the jsonic integration uses rule-state flags (`K["aql_tpl"]`) plus
dedicated tokens (`#BT`, `#IS`, `#TL`) inside the same parser. The
TSX-flavoured format extends the same approach with two additional
rules:

- A `tree` rule that opens on `<` and accepts elements, fragments,
  text, and `{...}` expression children.
- An `attr` rule that accepts string-literal or `{...}` expression
  values for attribute right-hand sides.

Both rules pivot back into the regular `val` rule on `{`, and
return on the matching `}`. Existing jsonic infrastructure handles
the bookkeeping.


## 4. Querying the form — CSS selectors

XPath has been the historical default for XML query. CSS selectors
are a more attractive choice for AQL:

- The selector vocabulary (descendant, child, attribute, `:nth-child`,
  etc.) is what most working programmers already know from the
  browser.
- Selector strings are far shorter than equivalent XPath for the
  common cases (`.foo > .bar` vs `descendant::*[contains(@class,
  "foo")]/child::*[contains(@class, "bar")]`).
- The selector engine is easier to implement and keep small than a
  full XPath 3.1 evaluator.

### 4.1 Prior art

| Library              | Language    | What it offers                                                                  |
|----------------------|-------------|---------------------------------------------------------------------------------|
| **Cheerio**          | Node.js     | jQuery-style API over parse5 / htmlparser2; the de facto server-side scraper.   |
| **jQuery / Sizzle**  | Browser JS  | The selector engine that popularised the pattern.                               |
| **BeautifulSoup**    | Python      | `.select()` accepts CSS; `.find_all()` accepts predicates; both compose freely. |
| **lxml.cssselect**   | Python      | Compiles CSS to XPath internally — proves the equivalence and lets users pick.  |
| **Nokogiri**         | Ruby        | Both `.css()` and `.xpath()` on the same DOM, with implicit conversion.         |
| **Goquery**          | Go          | jQuery-style chainable API on top of `golang.org/x/net/html`.                   |
| **Scrapy / Parsel**  | Python      | XPath and CSS, plus chained `.get()`/`.getall()` and `re()` for regex on hits.  |

### 4.2 Notable design features worth borrowing

From the survey:

- **Chained, composable accessors.** Cheerio's
  `$('div.x').children('.y').first()` reads as a pipeline; AQL's
  concatenative shape is naturally pipeline-like, so a CSS selector
  word fits naturally.
- **Mixed selector types.** BeautifulSoup lets you pass a CSS
  selector, an attribute dict, or a predicate function to the same
  `find_all`. The same arg signature in AQL
  (`Selector | Map | FnDef`) gives users freedom without bifurcating
  the API.
- **Pseudo-classes for shape, not state.** `:has(...)`, `:not(...)`,
  `:contains(...)` are all data-shape predicates that translate
  cleanly to AQL filter words. Browser-only pseudos (`:hover`,
  `:focus`) are dropped because they have no analogue in static
  documents.
- **Result is itself a queryable collection.** Goquery returns a
  `*Selection` which is itself selectable. AQL should make selector
  results lists of node values that accept further selector words as
  suffixes — chaining falls out of the existing forward-precedence
  rule.
- **Regex post-filter.** Parsel's `re()` and Scrapy's `.re_first()`
  show that regex composition on selector output is a frequent need;
  pairing CSS selection with `rm/.../` (see `MINILANG.0.md`) closes
  the loop without a new word.
- **Compile-once selectors.** Sizzle and Goquery both expose
  precompiled forms. AQL's `cs/` mini-language literal is already a
  compiled value, so this falls out for free.
- **`is`/`not` for predicate use.** jQuery's `$el.is('.foo')` is the
  same selector grammar reused as a predicate. AQL can do this with
  a single word that dispatches on a `node + cs/...` signature.

### 4.3 Surface for AQL

Reuse the `cs/` mini-language prefix introduced in `MINILANG.0.md`:

```
"<root><a class='x'>1</a></root>" cs/a.x
```

For multi-step queries, the result is a list and the suffix
collection rule applies normally:

```
xml-doc cs/article cs/h1 first
```

Companion words:

- `xml-attr` — take a node and an attribute name, return the
  attribute value (or `none`).
- `text` — extract concatenated text content of a subtree.
- `xml-is` — predicate version of `cs/`; tests whether a node
  matches the selector.

These combinators cover the common cases with vocabulary AQL users
already know.


## 5. Document-level features

The format carries some affordances that the bare token stream
cannot express:

- **Module imports** via the root `<aql>` element's attributes —
  e.g. `<aql import="aql:math">` — to scope a default module
  without an explicit `(module ...)` prelude.
- **Comments.** Both `<comment>...</comment>` siblings and
  JSX-style `{/* ... */}` inline comments are recognised. The
  concatenative form discards these, but the converter preserves
  them when round-tripping with the `--comments` option.
- **Boolean attribute shorthand.** `<def name="x" export>` is
  equivalent to `<def name="x" export={true}>`, matching JSX.
- **Source-position attributes** — `line`, `col`, `src` — for
  tools that diff between concrete syntaxes.

The format is otherwise a transparent encoding of the token stream;
a converter `aql-from-tsx` and `aql-to-tsx` round-trips between
concatenative and tree forms losslessly except for whitespace and
comments unless they are explicitly preserved by the converter
options.


## 6. Embedding other syntaxes — `<aql-embed>`

Because the format is a tree with attributes, any node can carry
foreign-syntax payload as element content. The `<aql-embed>` element
makes this explicit:

```tsx
<aql-embed lang="markdown">
  *foo* — a piece of inline markdown.
</aql-embed>

<aql-embed lang="sql">
  SELECT id, name FROM users WHERE active = true
</aql-embed>

<aql-embed lang="regex">
  ^[a-z][a-z0-9_]*$
</aql-embed>
```

The `lang` attribute names a registered embedding handler. Handlers
parse the body and produce an AQL value (typically a `MiniLang`
subtype, a string, a record, or a deferred parse-result). Unknown
`lang` values are surfaced as opaque strings carrying the body and
the language tag, so a document with unfamiliar embeddings still
loads.

This is a generalisation of the mini-language facility
(`MINILANG.0.md`): the `xy/` prefix gives a compact inline form for
short literals, and `<aql-embed>` gives a multi-line block form for
documents and longer payloads. The same handlers back both surfaces
— `cs/foo > .bar` and `<aql-embed lang="css">foo > .bar</aql-embed>`
produce the same value type.

### 6.1 Why this matters

- Documents (technical specs, runbooks, notebooks) commonly mix
  prose, code, queries, schemas, and config. A single AQL document
  can carry all of them through `<aql-embed>` without spawning N
  satellite files.
- The `lang` attribute is the natural extension point: new DSLs are
  added by registering a handler, not by modifying the parser.
- Tooling — editors, linters, formatters — can dispatch on `lang`
  to give each block its native treatment (syntax highlighting,
  autocompletion, formatter) while still treating the surrounding
  document as a single AQL program.

### 6.2 Interaction with interpolation

Embedded blocks may opt in to TSX-style `{...}` interpolation:

```tsx
<aql-embed lang="sql" interpolate>
  SELECT * FROM users WHERE id = {user_id}
</aql-embed>
```

The boolean attribute `interpolate` (JSX shorthand for
`interpolate={true}`) enables `{...}` substitution inside the body.
The lexer pivots into expression mode at `{` and the embedded
handler receives the post-substitution text — or, for handlers that
want it, the original template plus a list of substitutions, so a
SQL handler can produce a parameterised query with bind variables
rather than string concatenation.

Without the attribute, `{...}` is literal content of the embedded
language — the same convention TSX uses for `<pre>` and other
verbatim blocks. This matters for languages where `{` is meaningful
(C-family code, JSON, regex), which can be embedded without escape
churn:

```tsx
<aql-embed lang="json">
  { "id": 1, "name": "alice", "tags": ["a", "b"] }
</aql-embed>
```

### 6.3 Whitespace and indentation

Many embedded languages (Python, YAML, Markdown code fences) are
whitespace-sensitive. `<aql-embed>` honours an `xml:space="preserve"`
hint and supports a `dedent` boolean shorthand that strips the
common leading indent — the equivalent of Python's `textwrap.dedent`.
The default policy preserves content verbatim.


## 7. Status and open questions

This is a design sketch (completeness 0). No parser, converter, or
query engine work has started. Main open questions:

- Whether to mirror jsonic's grammar mechanics (extending the `val`
  rule with `tree` and `attr` peers) or build a separate TSX parser
  pass that emits the same engine `Value` stream.
- Round-tripping policy for comments and whitespace — match
  Prettier's TSX output, or preserve verbatim?
- Whether `<aql-embed>` blocks should be lazily parsed (handler
  invoked on first use) or eagerly parsed at load time.
- The exact tag set for structural primitives — the proposal here
  is illustrative, not final.
- Whether to reuse JSX's source-extension convention (`.aqx`?
  `.aqxs`?) for tooling discrimination, or treat the format as a
  surface variant of regular `.aql` files distinguished by
  content.
- Schema validation (RNG, XSD, or a bespoke AQL-flavoured schema)
  for tooling that wants structural guarantees.

Subsequent revisions will pin these down and bump the completeness
suffix in the filename.
